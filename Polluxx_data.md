# 🧬 OMNY-Twin (Polluxx) 数据引擎与极简特征管线
*Data Engine & Feature Processing Pipeline*

在 OMNY-Twin 处理 8,700 万条（包含超 4,000 万真实病例）海量医疗记录的过程中，我们采用了一套极简、高密度的特征预处理与装箱管线，彻底消灭了序列浪费与状态毒化。

---

## 1. 总体字典与基础标记 (Vocabulary & Layout)

整个模型被约束在一个精简的 `26,880` 维度词表中（对齐 64/128 倍数以完美迎合 GPU Tensor Core 优化）。

- **静止维度 (Static Tokens)**：针对每位病患的年龄、性别等静态人口学特征，在序列头部固定分配 **5 个 Token**。
- **动态维度 (Clinical Tokens)**：共包含近 26,868 个实体医疗编码，全面覆盖临床诊断 (Dx)、体征 (Vs)、实验室检验 (Lb) 和处方用药 (Rx)。
- **排版标记 (Special Tokens)**：
  - 病患级：`<patient_start>`, `<patient_end>`
  - 就诊级：`<enc_start>`, `<enc_end>`

---

## 2. 连续时序融合 (Continuous Time Positional Encoding)

我们**彻底废弃了**传统纵向上粗粒度的“离散时间占位符”（如用 `<gap_31-90>` 代表间隔 31 到 90 天），全面升级为**真实数值的时间特征融合**。

1. **提取**：读取两次相邻就诊之间绝对真实的浮点天数差（如 `time_delta = 42.5` 天）。
2. **编码**：利用 `ContinuousTimeEncoder`（基于 MLP 或正余弦函数），将 `42.5` 直接投影生成一个 1280 维的时序特征向量。
3. **加性融合**：不占用任何额外的序列位置！该 1280 维的时间特征会**直接叠加 (Add)** 到本次就诊记录下所有实体 Token 的 Embedding 上。

> **🌟 收益**：不仅极大拓展了 4096 上下文窗口能装载的就诊次数，同时让每一次诊断都具备了无限精细的历史时间距离感知。

---

## 3. 同次就诊的强制结构化 (Canonical Ordering)

面对单次看诊（Time = 0）内并发开出的海量无序事件（例如同一天开出 3 种药和 2 种化验），如果任其随机排列，会导致 Teacher Forcing 机制产生困惑，损失梯度归零。

**🛠️ 我们的规则：强制结构化降维**
所有的 `Intra-Encounter Tokens` 必须在预处理切词时，通过内部定序立规矩：
1. **大类按医学逻辑绝对锁死**：诊断 (dx) ➔ 体征 (vs) ➔ 检验 (lb) ➔ 药物 (rx)
2. **同类按字典序绝对排列**：例如两款药物同时开出时，阿司匹林一定排在某首字母靠后的药物前。

> **🌟 收益**：平息了由于“同义不同序”引发的自回归模型惩罚，模型在推断时自动还原出极度顺滑、规律的病历书写逻辑。

---

## 4. 极致装箱与状态防毒 (Sequence Packing)

面对病历长短极度不均的问题，我们禁止在短序列尾部使用 `<PAD>` 补齐，以防止 Mamba 历史状态被无意义的空白符“毒化”（Padding Poisoning）。

- **强行拼接 (Packing)**：将患者 A、患者 B、患者 C 的序列首尾相连，如贪吃蛇般直接塞满 `4096` 长度的训练窗口。
- **块对角掩码 (Block Diagonal Causal Mask)**：在 Attention 层通过该掩码构建物理防线，防止患者 B “看到” 患者 A 的病历。
- **强制遗忘 (State Reset)**：底层 Mamba 引擎在每一次滑过 `<patient_start>` 标记时，强制触发**连续时间编码与 Position ID 清零**，实现了内存级别且防反向传播干扰的病患隔离。
